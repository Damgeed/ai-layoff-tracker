## 2. CSS / Design System Audit

### 2.1 prefers-reduced-motion

**Two implementations found — excellent coverage:**

**timeline.css** (lines 66-77):
```css
@media (prefers-reduced-motion: reduce) {
  .entry-card.animate-in {
    opacity: 1;
    transform: none;
    transition: none;
  }
  .entry-card[data-index] { transition-delay: 0s !important; }
}
```

**main.css** (lines 622-629):
```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.001ms !important;
    transition-duration: 0.001ms !important;
  }
  html { scroll-behavior: auto; }
  .counter-value { animation: none !important; }
}
```

**JS charts.js** (lines 24-25, 63-65):
```js
const prefersReducedMotion = window.matchMedia('(prefers-reduced-motion: reduce)').matches;
// ... skips animation when true
```

✅ **Excellent.** Triple coverage: CSS global, CSS targeted, and JS. This exceeds WCAG AA requirements.

### 2.2 focus-visible Styles

```css
:focus-visible {
  outline: none;
  box-shadow: var(--focus-ring); /* 0 0 0 3px rgba(255, 68, 68, 0.4) */
  border-radius: var(--radius-sm);
}
a:focus-visible, button:focus-visible, input:focus-visible, select:focus-visible {
  outline: none;
  box-shadow: var(--focus-ring);
}
```

✅ **Good.** Uses `:focus-visible` (not `:focus`) to avoid showing focus rings on mouse clicks. The red glow ring is high-contrast and visually distinctive.

⚠️ **Minor:** `.filter-chip` buttons use `border-radius: 999px` (pill shape) but the `:focus-visible` uses `var(--radius-sm)` (4px). The square focus ring on a pill button looks slightly misaligned.

### 2.3 Mobile Breakpoints

| Breakpoint | Purpose | Status |
|------------|---------|--------|
| `@media (max-width: 1024px)` | Tablet — 2-col grids | ✅ |
| `@media (max-width: 768px)` | Mobile — stacked layouts | ✅ |
| `@media (max-width: 480px)` | Small mobile — single column | ✅ |
| `@media (min-width: 769px)` | Desktop timeline line | ✅ |

All required breakpoints (1024, 768, 480) are present. Responsive grid adaptations:
- Stats: 4-col → 2-col (1024) → 2-col (768) → 1-col (480)
- Charts: 3-col → 2-col (1024) → scroll-snap carousel (768)
- Downloads: 4-col → 2-col (1024) → 1-col (768)

### 2.4 Print Styles

```css
@media print {
  body { background: #fff; color: #000; }
  .site-header, .hero-actions, .filters-bar, .share-overlay, .site-footer { display: none; }
  .entry-card { border: 1px solid #ccc; break-inside: avoid; }
}
```

✅ **Present and sensible.** Hides interactive UI, inverts to light background, prevents card breaks.

### 2.5 Mobile Navigation Issue

⚠️ **Critical:** At `≤768px`, the header navigation is completely hidden:
```css
.site-header nav { display: none; }
.site-header .container { justify-content: center; }
```

**There is no hamburger menu, no alternative navigation mechanism.** Mobile users cannot access:
- Methodology page
- API endpoint
- CSV download
- RSS feed

This fails **WCAG 2.4.5 Multiple Ways** and **2.4.1 Bypass Blocks**.

**Severity: CRITICAL**

## 4. Mobile Responsiveness Deep-Dive

### 4.1 Viewport & Meta Tags

```html
<meta name="viewport" content="width=device-width, initial-scale=1.0">
```
✅ Present on all pages. `-webkit-text-size-adjust: 100%` in CSS.

### 4.2 Touch Target Sizes

| Element | Size | WCAG 2.5.5 (44×44px)? |
|---------|------|----------------------|
| `.btn` | `0.625rem + padding` (~38px height) | ⚠️ Slightly under |
| `.filter-chip` | `0.375rem padding` (~24px height) | ❌ **Too small** |
| `.filter-toggle` | `0.5rem padding` (~29px height) | ❌ **Too small** |
| Nav links | ~20px height | ❌ **Too small** |
| Search input | `0.75rem padding` (~44px height) | ✅ |

**Severity: MEDIUM** — Filter chips and mobile filter toggle fall below 44px touch target.

### 4.3 Horizontal Scroll

Charts section at `≤768px` uses horizontal scroll-snap:
```css
.charts-grid {
  overflow-x: auto;
  scroll-snap-type: x mandatory;
}
```
✅ This is a valid responsive pattern and works well on mobile.

### 4.4 Content at 320px Width

The narrowest supported width is 320px (iPhone SE). At 480px, everything stacks to single column. At 320px:
- `.container { padding: 0 0.75rem; }` — ✅ adequate
- Counter value uses `clamp(3.5rem, 8vw, 6rem)` — ✅ scales down
- Entry cards `flex-direction: column` — ✅ stacks correctly

### 4.5 Mobile-Specific Issues

1. **No hamburger menu** — Navigation is simply hidden. This is the #1 mobile UX issue.
2. **Filter toggle button** `<button class="filter-toggle">⚙ Filters</button>` lacks `aria-expanded` state update in JS. The initial value `aria-expanded="false"` is set but `toggleFilter()` in app.js only toggles `.open` class on `.filters-bar` without updating `aria-expanded`.
3. **Chart carousel** lacks left/right scroll indicators or "swipe" hint.
4. **Share overlay on mobile** — The 600px max-width modal with share image may overflow on small screens.

## 5. JS-Generated Content Accessibility

### 5.1 Dynamically Rendered Cards (app.js)

Each entry card is an `<article>` element with proper structure. Generated elements include `aria-label` on share buttons:
```js
aria-label="Share ${escAttr(entry.company)}"
```
✅ Good.

### 5.2 Missing ARIA Updates

- **No `aria-live` announcement when timeline loads/refreshes** — screen readers don't know content changed.
- **No focus management after filter application** — focus stays on the filter control even though results have updated.
- **No `role="status"` on the toast notification** — `.toast` appears without screen reader announcement.

### 5.3 Charts Canvas Accessibility

Each canvas has `role="img"` and an `aria-label`:
```html
<canvas id="chart-industry" role="img" aria-label="Bar chart of jobs impacted by industry"></canvas>
```

However, these labels are **static** — they don't update when data changes. There are no hidden `<table>` fallbacks or detailed `aria-describedby` summaries. This is acceptable for decorative/presentational charts but limits data understanding for screen-reader users.

**Severity: LOW** — Charts are supplementary to the text-based timeline.

## 9. Methodology

This audit was conducted via static code analysis of the complete codebase:

- **17 HTML files** — All pages manually inspected for ARIA attributes, semantic structure, heading hierarchy, and keyboard accessibility.
- **2 CSS files** (1,377 combined lines) — Analyzed for breakpoints, print styles, motion preferences, focus styles, and design tokens.
- **5 JS files** (1,563 combined lines) — Reviewed for dynamic ARIA updates, focus management, and reduced-motion support.
- **Contrast ratios** — Calculated using the WCAG 2.1 relative luminance formula:
  - Linearize sRGB: `L = 0.2126×R + 0.7152×G + 0.0722×B`
  - Contrast: `(L1 + 0.05) / (L2 + 0.05)`
- **Effective background for badges** — Computed alpha-blending over `#141414` surface.

## 10. Conclusion

The `index.html` and `methodology.html` pages demonstrate strong accessibility awareness — skip links, proper ARIA, `aria-live` regions, focus management, and keyboard shortcuts exceed WCAG AA baseline in many areas. The CSS design system is well-crafted with `prefers-reduced-motion`, `:focus-visible`, three mobile breakpoints, and print styles.

However, the **15 company pages are a significant regression** — they lack nearly all accessibility features present on the main pages. This appears to be a templating gap: fixes applied to the main templates were not backported.

**Two contrast failures** (primary button text, muted body text) affect ubiquitous elements and should be addressed with simple color value changes.

The **missing mobile navigation** is the single most impactful issue — at `≤768px`, users have no way to navigate between pages.

**Overall:** The site is 74% WCAG AA compliant. With the prioritized fixes above, it can reach 95%+ compliance within a few hours of work.
