Corrections

We are committed to accuracy. If you find an error in the database, here is how to report it and what to expect.

How to Report an Error

To submit a correction, follow these steps:

  1. Identify the entry ID. Each entry in the database has a unique ID (e.g., klarna-2024-08). You can find this in the entry's URL or in the JSON data under the id field.
  2. Describe the error. Explain what is incorrect and what the correct information should be. Be as specific as possible — include data points, dates, and figures.
  3. Provide a source URL. Include at least one publicly accessible source that supports the correction.
  4. Email us. Send your correction to corrections@ailayofftracker.com with all of the above information.

Evidence Requirements

All correction submissions must meet the following evidentiary standards:

  • Primary source required. The correction must be supported by at least one primary source — a company press release, SEC filing, earnings call transcript, verified media report, or official regulatory document.
  • Publicly accessible URL. The source must be accessible without login, paywall exceptions apply for major news organizations.
  • Directly contradicts existing entry. The source must clearly and specifically contradict the current entry data. General context or opinion pieces are not sufficient.
  • Verifiable by our team. We will independently verify the source before accepting the correction.

Review Process

Every correction submission follows a standardized review workflow:

StageDescriptionTimeline
1. Acknowledgment We confirm receipt of your correction and assign it a tracking identifier. You will receive an automated acknowledgment email. Within 72 hours
2. Source Verification Our review team locates and verifies the supporting source. We cross-reference the claimed correction against the original entry sources and any additional corroborating material. 1–5 business days
3. Decision A decision is made based on the evidence. Possible outcomes: ACCEPT (correction applied, entry updated, changelog published), REJECT (evidence does not support the correction, explanation provided), or HOLD (pending additional evidence from submitter or team). 1–3 business days after verification
4. Publication If accepted, the corrected entry is published with an update timestamp and a changelog entry documenting what was changed, when, and why. All corrections are logged in our public changelog. Immediate upon acceptance

Decision Outcomes

ACCEPT — The correction has been verified and applied. The entry now reflects the corrected data, and the changelog records the change.

REJECT — The submitted evidence does not support the correction. The entry remains unchanged. A detailed explanation is provided to the submitter.

HOLD — The correction cannot be verified with the provided evidence alone. Additional sourcing is needed. The submitter is invited to provide further documentation.

Response Times

We aim to process all correction submissions promptly:

  • Initial acknowledgment: Within 72 hours of submission.
  • Resolution (accepted or rejected): Typically 1–2 weeks, depending on the complexity of the correction and availability of corroborating sources.
  • Complex cases: Corrections involving multiple data points, ambiguous source language, or tier reclassification may take up to 3 weeks.
  • Expedited review: Corrections for factual errors with immediate public significance (e.g., incorrect job-loss figures in active news coverage) are prioritized.

If you have not received a response within the stated timeframe, please follow up at corrections@ailayofftracker.com.

Corrections JSON Schema

All correction submissions are structured as JSON objects conforming to the following schema. This schema is also available at /api/corrections-schema.json.

{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "title": "Correction Submission",
  "type": "object",
  "properties": {
    "entry_id": {
      "type": "string",
      "description": "ID of the entry being corrected"
    },
    "reporter_email": {
      "type": "string",
      "format": "email"
    },
    "reported_date": {
      "type": "string",
      "format": "date"
    },
    "error_type": {
      "type": "string",
      "enum": [
        "DATA_ERROR",
        "CLASSIFICATION_ERROR",
        "SOURCE_ERROR",
        "OUTDATED"
      ]
    },
    "description": {
      "type": "string",
      "minLength": 20
    },
    "supporting_source_url": {
      "type": "string",
      "format": "uri"
    },
    "status": {
      "type": "string",
      "enum": [
        "PENDING",
        "ACCEPTED",
        "REJECTED"
      ]
    },
    "resolution_date": {
      "type": "string",
      "format": "date"
    },
    "resolution_notes": {
      "type": "string"
    }
  },
  "required": [
    "entry_id",
    "reporter_email",
    "error_type",
    "description",
    "supporting_source_url"
  ]
}

Field Reference

FieldTypeRequiredDescription
entry_idstringYesID of the entry being corrected (e.g., klarna-2024-08)
reporter_emailstring (email)YesYour email address for follow-up correspondence
reported_datestring (date)NoDate the error was identified (ISO 8601)
error_typestring (enum)YesOne of: DATA_ERROR, CLASSIFICATION_ERROR, SOURCE_ERROR, OUTDATED
descriptionstringYesDetailed description of the error (minimum 20 characters)
supporting_source_urlstring (uri)YesPublicly accessible URL supporting the correction
statusstring (enum)NoReview status: PENDING, ACCEPTED, or REJECTED (set by reviewer)
resolution_datestring (date)NoDate the correction was resolved (ISO 8601, set by reviewer)
resolution_notesstringNoNotes explaining the resolution outcome (set by reviewer)

Error Types

  • DATA_ERROR — Factual error in entry data (incorrect job-loss count, wrong date, incorrect company name, etc.).
  • CLASSIFICATION_ERROR — Entry is assigned to the wrong tier or has incorrect evidence type tags.
  • SOURCE_ERROR — Cited source does not support the entry claims, link is broken, or source is misattributed.
  • OUTDATED — Entry is factually outdated due to new information or subsequent developments.
